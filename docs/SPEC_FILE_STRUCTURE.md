# Armature Spec File Structure

Reference for creating, organizing, and using spec files with Armature's budget-controlled harness.

---

## Directory Layout

```
your-project/
  armature.yaml              # project config (specs: block enables spec system)
  specs/
    SPEC-2026-Q2-001.yaml    # feature spec
    SPEC-2026-Q2-002.yaml    # bugfix spec
    SPEC-2026-Q2-003.yaml    # refactor spec
    templates/
      feature.yaml           # template for new features
      bugfix.yaml            # template for bug fixes
      refactor.yaml          # template for refactoring
      spike.yaml             # template for investigations
  .armature/
    baselines/
      SPEC-2026-Q2-001.json  # quality baseline (auto-generated by pre-dev)
    budget/
      SPEC-2026-Q2-001_cost.jsonl  # token/cost log (auto-generated)
```

## Enabling Specs in armature.yaml

```yaml
specs:
  enabled: true
  dir: "specs/"
  templates_dir: "specs/templates/"
  id_pattern: "^SPEC-\\d{4}-Q[1-4]-\\d{3}$"
  required_fields:
    - spec_id
    - title
    - type
    - description
    - acceptance_criteria
  traceability:
    enabled: true
    pattern: "(SPEC-\\d{4}-Q\\d-\\d{3})\\s*/\\s*(AC-\\d+)"
```

## Spec ID Format

Pattern: `SPEC-{YEAR}-Q{QUARTER}-{SEQUENCE}`

| Part | Format | Example |
|------|--------|---------|
| Year | 4 digits | 2026 |
| Quarter | Q1-Q4 | Q2 |
| Sequence | 3 digits | 001 |

Full example: `SPEC-2026-Q2-001`

The ID is used as:
- File name: `specs/SPEC-2026-Q2-001.yaml`
- Budget tracking key: `.armature/budget/SPEC-2026-Q2-001_cost.jsonl`
- Baseline key: `.armature/baselines/SPEC-2026-Q2-001.json`
- CLI argument: `armature pre-dev SPEC-2026-Q2-001`
- Test traceability: `SPEC-2026-Q2-001 / AC-1` in test docstrings

## Spec File Format (YAML)

### Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `spec_id` | string | Unique identifier matching `id_pattern` |
| `title` | string | Short human-readable title |
| `type` | string | One of: `feature`, `bugfix`, `refactor`, `spike` |
| `description` | string | What needs to be done and why |
| `acceptance_criteria` | list | Testable conditions with `id` and `description` |

### Optional Fields

| Field | Type | Description |
|-------|------|-------------|
| `priority` | string | `low`, `medium`, `high`, `critical` |
| `author` | string | Who wrote the spec |
| `date` | string | ISO date (YYYY-MM-DD) |
| `scope.modules` | list | Affected source directories |
| `scope.touches_api` | bool | Whether API surface changes |
| `scope.touches_ui` | bool | Whether UI changes |
| `scope.new_files_expected` | list | Files to be created |
| `scope.modified_files_expected` | list | Files to be modified |
| `constraints` | list | Non-functional requirements, limits |
| `human_gates` | list | Approval checkpoints with `gate`, `approver`, `description` |
| `depends_on` | list | Spec IDs this depends on |
| `blocks` | list | Spec IDs blocked by this |
| `eval.unit_test_coverage_min` | int | Minimum coverage percentage |
| `eval.integration_test_required` | bool | Whether integration tests needed |
| `eval.e2e_test_required` | bool | Whether E2E tests needed |
| `eval.linting_must_pass` | bool | Lint gate |
| `eval.type_check_must_pass` | bool | Type check gate |

### Full Example

```yaml
# specs/SPEC-2026-Q2-001.yaml
spec_id: "SPEC-2026-Q2-001"
title: "Add user authentication endpoint"
type: feature
priority: high
author: "developer"
date: "2026-04-15"

description: |
  Add JWT-based authentication to the API. Users should be able to
  register, login, and receive access tokens. Protected routes
  require a valid token in the Authorization header.

scope:
  modules: ["app/routes", "app/services"]
  touches_api: true
  touches_ui: false
  new_files_expected:
    - "app/routes/auth.py"
    - "app/services/auth_service.py"
  modified_files_expected:
    - "app/routes/__init__.py"

acceptance_criteria:
  - id: AC-1
    description: "POST /register creates a new user"
    testable: true
  - id: AC-2
    description: "POST /login returns a JWT token"
    testable: true
  - id: AC-3
    description: "Protected routes return 401 without valid token"
    testable: true

constraints:
  - "Use PyJWT for token generation"
  - "Passwords hashed with bcrypt"
  - "Token expiry: 24 hours"

human_gates:
  - gate: "plan_review"
    approver: "developer"
    description: "Review implementation plan"
  - gate: "code_review"
    approver: "developer"
    description: "Review generated code"

depends_on: []
blocks: ["SPEC-2026-Q2-003"]

eval:
  unit_test_coverage_min: 90
  integration_test_required: true
  e2e_test_required: false
  linting_must_pass: true
  type_check_must_pass: true
```

## Spec Templates by Type

| Type | Required Sections | Optional |
|------|-------------------|----------|
| **feature** | description, scope, acceptance_criteria, constraints, human_gates, eval | depends_on, blocks |
| **bugfix** | description, root_cause, reproduction_steps, acceptance_criteria, eval | scope |
| **refactor** | description, scope, constraints, eval | acceptance_criteria |
| **spike** | question, scope, time_box, deliverable | eval |

## Test Traceability

Link acceptance criteria to tests using the traceability pattern in test docstrings:

```python
def test_register_creates_user():
    """SPEC-2026-Q2-001 / AC-1"""
    response = client.post("/register", json={"email": "a@b.com", "password": "secret"})
    assert response.status_code == 201
```

```typescript
test("login returns JWT token", () => {
  // SPEC-2026-Q2-001 / AC-2
  const response = await fetch("/login", { method: "POST", body: credentials });
  expect(response.json().token).toBeDefined();
});
```

Armature traces these references to verify every AC has at least one test.

## Workflow Integration

```
1. Write spec           specs/SPEC-2026-Q2-001.yaml
2. Pre-dev baseline     armature pre-dev SPEC-2026-Q2-001
3. Develop              (agent writes code, armature checks on each file write)
4. Budget tracking      armature budget report SPEC-2026-Q2-001
5. Post-dev regression  armature post-dev SPEC-2026-Q2-001
6. Traceability check   verify AC coverage in tests
```

## Ossature Compatibility

If converting from ossature specs (`.smd`/`.amd` files):

```bash
armature compat convert /path/to/ossature-project --output specs/converted.yaml
```

The converter maps ossature `@id`, `@priority`, `@depends` directives to armature spec fields, and `.amd` architecture components to `architecture.layers` and `architecture.boundaries`.
