# AI Context Engineering — Unique Feature Deep Dive

**Armature is the only framework that treats AI agent context as a governable resource.** Every token in the context window costs money and affects quality. Context engineering optimizes what the agent sees, when, and how — turning a 200K context window from a liability into a precision instrument.

---

## The Problem

AI coding agents waste 30-60% of their context window on irrelevant code:

```
Without context engineering:
  Agent reads 50 files "just in case" → 150K tokens consumed
  Agent misses the 3 files that matter → wrong solution
  Agent hits context limit → loses early context
  Cost: $0.45 per request (at Sonnet pricing)

With Armature context engineering:
  Agent reads 8 targeted files → 25K tokens consumed
  Progressive disclosure surfaces relevant code first
  Context stays focused throughout session
  Cost: $0.075 per request (6x cheaper, better results)
```

## Feature 1: CLAUDE.md / AGENTS.md Generation

Armature generates a project context file that AI agents read at session start — giving them architecture rules, quality standards, and project conventions without consuming tokens on irrelevant files.

### What Gets Generated

```markdown
# CLAUDE.md (auto-generated by Armature)

## Architecture
- 6 layers: _internal, config, quality, budget, architecture, heal
- _internal and config are shared (any layer can import)
- CLI and MCP are entry points — never import from each other

## Boundaries (ENFORCED)
- cli/ must NOT import from: budget/, quality/, heal/ directly
- quality/ must NOT import from: cli/, mcp/
- budget/ must NOT import from: cli/, mcp/, quality/

## Quality Gates
- draft: 70% weighted score
- review_ready: 85%
- merge_ready: 95%

## Checks
Run `armature check` before committing. 8 weighted checks:
lint (W:25), type_check (W:25), test (W:20), security (W:20),
complexity (W:15), dependency_audit (W:15), test_ratio (W:10), docstring (W:10)

## Self-Healing
Run `armature heal --failures lint` to auto-fix lint violations.
Circuit breaker opens after 3 consecutive failures — escalate to human.

## Budget
Token budget tracked per spec. Circuit breaker at 3 consecutive overruns.
Phase allocation: validate 5%, audit 10%, plan 15%, build 40%, test 25%, review 5%
```

### Configuration

```yaml
# armature.yaml
context:
  enabled: true
  agent_md:
    path: "CLAUDE.md"          # output file name
    max_lines: 200              # keep context compact
    deep_docs:                  # additional files to reference
      - "docs/ARCHITECTURE.md"
      - "docs/API.md"
  progressive_disclosure: true
  memory_dir: ".claude/memory/"
```

## Feature 2: Progressive Disclosure

Instead of dumping the entire codebase into context, Armature surfaces information in layers — starting with what's most likely relevant.

### Disclosure Layers

```
Layer 1 (Always loaded — ~500 tokens):
  CLAUDE.md → architecture rules, quality gates, conventions

Layer 2 (On task start — ~2K tokens):
  armature.yaml → project config, check settings, budget tiers
  Spec file → acceptance criteria, constraints for this task

Layer 3 (On demand — variable):
  Relevant source files → only files touched by the spec
  Test files → only tests covering the changed modules
  Deep docs → architecture docs, API docs as needed

Layer 4 (Never loaded — saved tokens):
  Build artifacts, node_modules, .git history
  Unrelated modules, configuration for other environments
  Historical specs that aren't relevant to current task
```

### Token Estimation

Before reading a file, Armature estimates its token cost:

```python
# Approximate: 1 token ≈ 4 characters for code
def estimate_tokens(file_path: Path) -> int:
    content = file_path.read_text()
    return len(content) // 4

# Example: should we read this 500-line file?
# estimate_tokens("src/budget/calibrator.py") → ~4,800 tokens
# At Sonnet pricing: 4800 * $3/1M = $0.000014
# Decision: only read if spec touches budget calibration
```

### Narrow Scope Optimization

```yaml
# Armature narrows agent context based on spec:
#
# Spec: "Fix login timeout bug"
# Relevant: auth/, session/, middleware/
# Irrelevant: budget/, reporting/, docs/
#
# Without narrowing: agent reads 50 files (150K tokens)
# With narrowing: agent reads 8 files (25K tokens)
# Savings: 83% fewer tokens, 6x cheaper, better focus
```

## Feature 3: Cross-Session Memory

Armature maintains persistent memory across AI agent sessions — so the agent doesn't rediscover the same things every conversation.

### Memory Structure

```
.claude/memory/
├── user_role.md              # developer profile, expertise areas
├── feedback_testing.md       # "don't mock the database" — learned preference
├── project_auth_rewrite.md   # ongoing auth rewrite context
└── reference_linear.md       # "bugs tracked in Linear project INGEST"
```

### Memory Types

| Type | What It Stores | Example |
|------|---------------|---------|
| `user` | Developer role, expertise, preferences | "Senior Go engineer, new to React" |
| `feedback` | Corrections and validated approaches | "Don't use mocks in integration tests" |
| `project` | Ongoing work, decisions, deadlines | "Auth rewrite driven by legal compliance" |
| `reference` | External system pointers | "Bugs tracked in Linear project INGEST" |

### How Memory Saves Tokens

```
Session 1: Developer asks "explain the budget module"
  → Agent reads 12 files, builds understanding
  → Memory saves: "Budget module uses EMA calibration, 
     JSONL per-spec tracking, circuit breaker at 3 overruns"

Session 2: Developer asks "add a new budget report"
  → Agent recalls memory instead of re-reading 12 files
  → Saved: ~15K tokens ($0.045 at Sonnet pricing)
  → Agent jumps straight to implementation
```

## Feature 4: Semantic Cache Integration

Armature's semantic cache (in the budget pillar) also reduces context waste:

```yaml
budget:
  cache:
    enabled: true
    storage: ".armature/cache/"
    max_size_mb: 100
    ttl_days: 7
```

### How It Works

```
Request 1: "Explain the ModelRouter class"
  → Agent reads router.py (4K tokens), generates explanation
  → Response cached with structural fingerprint

Request 2: "How does model routing work?"
  → Semantic match to cached response (same structural fingerprint)
  → Cache hit: 0 tokens consumed, instant response
  → Savings: 4K input + 2K output tokens saved
```
